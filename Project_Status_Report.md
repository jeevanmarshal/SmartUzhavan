## 1. Executive Summary
The primary objective was to implement a robust PDF billing system capable of rendering **Tamil Unicode** text correctly while ensuring the text remains **selectable** (not image-based). After multiple attempts with frontend libraries, the system has been migrated to a **Node.js Backend Architecture** using **PDFKit** to ensure production-grade stability and font compliance.

## 2. Failure Logs & Debugging Journey

### Phase 1: jsPDF (Frontend)
- **Problem:** Tamil characters appeared as broken symbols (Mojibake) or boxes.
- **Critical Log:** `Uncaught Error: No unicode cmap for font`.
- **Root Cause:** jsPDF's internal font engine requires a complex Character Map (CMap) that was missing from the provided Base64 strings.
- **Debugging:** Attempted to inject Noto Sans Tamil via Base64, but the engine failed to calculate character widths, leading to a silent crash.

### Phase 2: html2pdf / html2canvas
- **Problem:** PDFs were generated as images.
- **User Feedback:** "The farmer's only understand the tamil text... but text is not selectable."
- **Root Cause:** These libraries "screenshot" the DOM. This makes text unsearchable and creates large file sizes unsuitable for professional billing.

### Phase 3: pdfmake (Frontend)
- **Problem:** Critical initialization failures in the Vite/React environment.
- **Failure Logs:**
    1. `Uncaught TypeError: Cannot read properties of undefined (reading 'vfs')`
    2. `Error: File 'NotoSansTamil.ttf' not found in virtual file system`
    3. `RangeError: Offset is outside the bounds of the DataView`
- **Debugging Discovery:** Upon deep analysis of the font files (e.g., `tamilFont.js`), we discovered they were only **81 bytes** in size.
- **The "Smoking Gun":** These files were empty placeholders containing only a header. When the PDF engine tried to parse them as real fonts, it crashed because the data was missing.

## 3. Bugs Resolved & Technical Fixes

| Bug ID | Error Message | Resolution |
| :--- | :--- | :--- |
| **B001** | `TypeError: pdfMake.createPdf is not a function` | Resolved by switching from Namespace imports (`import * as`) to Default imports in the Vite environment. |
| **B002** | `Parameter 'options' has an invalid type` | Fixed by explicitly passing all four arguments to `createPdf` and using `{}` instead of `null` for table layouts. |
| **B003** | `readFileSync is not a function` | Caused by pdfmake trying to access the server filesystem from the browser. Resolved by moving generation to the Backend. |
| **B004** | `Offset outside bounds of DataView` | Implemented a **Font Safety Guard** that checks if a font file is > 10KB before loading, preventing server crashes from placeholder files. |

## 4. Final Architecture: Node.js + PDFKit

We have moved the heavy lifting to a dedicated **PDF Server**.

### Backend (Node.js)
- **Engine:** `PDFKit`.
- **Strategy:** Direct filesystem access to `.ttf` font files.
- **Logic:** Manual coordinate-based drawing for Harvester Bills, Rental Receipts, and Account Statements.
- **Result:** Native Unicode support, selectable text, and professional layouts.

### Frontend (React)
- **Service:** `pdfService.js` now acts as a thin wrapper.
- **Mechanism:** Sends JSON data to the backend via `fetch()` and downloads the resulting binary Blob as a `.pdf`.
